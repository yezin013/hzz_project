# Amplify 배포 실패 분석 - 404 에러

## 🚨 결론부터: **Amplify는 현재 프로젝트에 적합하지 않습니다**

네, **Amplify로는 배포가 불가능합니다.** (정확히는 SSR/인증이 필요한 현재 구조로는 불가능)

---

## 📊 무엇이 문제인가?

### ✅ 작동하는 것:
- 빌드 프로세스 (성공)
- 정적 페이지 (`/`, `/board`, `/info` 등)

### ❌ 작동하지 않는 것:
- **동적 라우트** (`/api/auth/[...nextauth]`, `/drink/[id]`)
- **NextAuth 로그인**
- **서버사이드 렌더링 (SSR)**

빌드 로그에는 이렇게 나옵니다:
```
ƒ /api/auth/[...nextauth] (Dynamic) ← SSR 라우트 감지됨
```

**하지만 실제 배포 시 404 에러!**

---

## 🔍 왜 안 되는가?

### **AWS Amplify Hosting Gen 1의 한계**

Amplify Hosting Gen 1은 다음 두 가지만 지원합니다:
1. **완전 정적 HTML** 사이트 (Static Export)
2. **Amplify Gen 2** 형식 (Lambda 완전 통합)

**우리 프로젝트는:**
- NextAuth로 Google 로그인 (API Routes 필요)
- 서버사이드 렌더링 사용
- 동적 라우팅 (`[id]` 등)

→ **Amplify Gen 1으로는 불가능!**

---

## 💡 해결책

### **옵션 1: K8s 사용 유지** ⭐⭐⭐ **강력 추천!**

**현재 상태:**
- ✅ 로컬 K8s: 완벽 작동
- ✅ AWS EKS: 완벽 작동
- ✅ 모든 기능 정상 (SSR, 인증, API)
- ✅ GitOps 자동 배포 (ArgoCD)

**비용:**
- K8s: ~30-40달러/월
- Amplify: ~30-50달러/월
- **차이 거의 없음!**

**결론:** 이미 완벽하게 작동하는데 굳이 Amplify로 바꿀 필요 없음!

---

### **옵션 2: Static Export로 변경** ❌ **비추천**

Amplify에 맞추려면 다음을 포기해야 합니다:
- ❌ 서버사이드 렌더링
- ❌ API Routes
- ❌ NextAuth (구글 로그인)
- ❌ 동적 콘텐츠

**즉, 완전히 정적인 사이트로만 만들어야 함**

→ 로그인 기능 자체가 불가능해짐!

---

### **옵션 3: Amplify Gen 2로 완전 마이그레이션** ⚠️ **복잡**

**필요 작업:**
- Amplify Gen 2 설정 완전 재작성
- Lambda 함수로 SSR 설정
- 추가 AWS 서비스 연동

**예상 시간:** 2-3일  
**복잡도:** 매우 높음  
**득실:** K8s가 이미 작동하는데 굳이?

---

## 🎯 최종 권장사항

### **Amplify 포기하고 K8s 사용!**

**이유:**

1. **K8s가 이미 완벽함**
   ```
   ✅ 로컬 개발: K8s
   ✅ 프로덕션: AWS EKS
   ✅ 자동 배포: GitLab CI/CD + ArgoCD
   ✅ 모든 기능 작동
   ```

2. **비용 차이 없음**
   - K8s도 서버리스만큼 저렴
   - Auto-scaling 설정 가능

3. **더 많은 제어권**
   - 완전한 커스터마이징
   - 백엔드와 같은 환경
   - 네트워크 효율성

4. **이미 구축 완료**
   - 추가 작업 불필요
   - 검증된 배포 파이프라인

## 🚀 결론

**Amplify는 이 프로젝트에 맞지 않습니다.**

- 현재 K8s 배포가 **프로덕션 레벨**로 완성되어 있습니다
- Amplify로 바꾸면 **기능을 잃거나** 엄청난 **추가 작업**이 필요합니다
- **비용 절감 효과도 없습니다**

**권장:** Amplify 앱 삭제하고 K8s 배포만 사용하세요!
